The stack

A pragmatic monorepo for a small team driving AI tooling. Nothing exotic — every piece is chosen because it's well-trodden and the failure modes are known.

apps/web

Next.js + TypeScript on Vercel. The CRM UI, the AI assist, the connect screens. Never imports a chat SDK.

apps/worker

An always-on Node service (Fly.io / Railway) with a Redis-backed job queue. Holds the live messaging sessions. The only code that touches a provider.

packages/shared

Types, database schema, validation, crypto, and the channel interface. No provider SDKs live here.

Data

Supabase Postgres + Row-Level Security + Auth. Transaction-pooled for the web app; a direct connection for the always-on worker.

The MessagingProvider abstraction

This one interface is what makes "lose a channel, not your business" true in code. WhatsApp (official), WhatsApp (unofficial fallback) and Telegram are three implementations of a single interface. The CRM only ever sees the interface — it never knows or cares which channel a message rode in on. Adding Signal later means writing one more implementation, not touching the CRM.

Tech info: the interface and the capability-driven UI For technical readers

Every provider exposes the same shape: connect(), getStatus(), send(), syncHistory(), disconnect(), and an onInbound() handler — plus a capabilities object the UI reads to render itself correctly without channel-specific branching:

  • connectMethod (qr / pair_code / bot_token) picks the right connect screen.
  • requires24hWindow shows or hides the "window closing — template only" banner.
  • historySyncDays / historySyncMode drive the "syncing history…" state.
  • readReceipts, groups, etc. hide UI a channel doesn't support.

Inbound messages are normalised into one shape, encrypted, then upserted idempotently by (conversation, provider_message_id) — so a backfill and a live message can never duplicate. Ordering is always by the provider's send time, never insert time. Outbound is claim-then-send with an idempotency key, consumed only from human-approved rows, so a worker restart can't double-send.

Why the messaging session can't be serverless

A WhatsApp or Telegram session is a persistent, always-open connection that must stay alive 24/7 — even while the worker's phone is off. Serverless platforms (like Vercel Functions) cap function runtime at a few minutes and explicitly can't act as a long-lived connection server. So the session layer lives in a dedicated, always-on process; the CRM stays serverless and talks to it only through the database. This split isn't a preference — it's a hard technical boundary, and it's why hosting the worker is a real (paid) line item.

A subtle but fatal gotcha

Free database projects pause after about 7 days idle — fatal for an always-needed messaging session. The worker and its Redis must be on paid, always-on hosting from day one. Budgeted for; see Costs.

Hard platform constraints we can't engineer away

Honesty about limits is part of the diligence. These are walls, not bugs:

ConstraintWhat it meansHow we live with it
24-hour window Free-form messages only within 24h of a client's last message; otherwise approved templates only. UI surfaces window state; keep replies inside the free window; templates for re-engagement.
History caps (Coexistence) ~180 days text, ~14 days media, no groups sync into the API. Full history stays on-device with the worker; the CRM uses a recent window.
Regional availability Coexistence is unavailable in many regions (EU/UK/AU/JP/KR commonly) and the list shifts. Verify per user country; fall back to Telegram or scoped unofficial where unavailable.
Per-portfolio number cap ~2 numbers initially, ~20 by default per business portfolio. Don't design one-number-per-worker at scale; pool shared numbers. See Scaling.
Portfolio-level enforcement Numbers grouped together share quality & limits (since Oct 2025) — one bad number can block the group. Isolate portfolios; treat it as a real single-point-of-failure.
No groups, presence leaks Group chats unsupported on the API; read receipts / typing can reveal automation timing. Hide unsupported features via capabilities; keep human-paced behaviour.

Source: docs/research/00_synthesis.md §3, §11; docs/CONTRACTS.md §2. All platform numbers are medium-confidence and revised by Meta periodically — re-verify before any launch.

Day-one security guardrails

Security here isn't a phase — it's switched on from the first commit, because the data is too sensitive for "we'll harden it later."

Secrets

Gitignored env files, a secrets manager, and an explicit deny-rule so the AI coding tool itself can't read .env — a gitignore alone does not stop that.

Leak detection

gitleaks as both a pre-commit hook and a CI job — the final net against a committed secret.

RLS everywhere

Row-Level Security on every table, deny-by-default. The build fails if any policy is a blanket allow-all.

No plaintext in logs

Message bodies, previews and raw payloads are redacted before logging; structured logs carry only ids, timestamps and counts.

Pinned dependencies

Provider libraries are version-pinned and lockfiles audited — direct response to the Dec 2025 poisoned-fork incident.

Real endpoint auth

Bridge endpoints authenticated by the user's verified token; any service secret compared in constant time. No static shared secrets doubling as authorization.

Tech info: the guardrails in specifics For technical readers
  • Claude Code deny rules — .claude/settings.json denies Read(./.env), Read(./.env.*), Read(./secrets/**). Confirmed (The Register, Jan 2026) that .gitignore/.claudeignore do not stop the tool reading .env — only the deny rule does.
  • RLS CI gate — the pipeline greps migrations and fails on WITH CHECK (true) or USING (true). Every policy constrains to the row's owner; service_role is worker-only, never in the browser.
  • Encryption — libsodium XChaCha20-Poly1305, per-user DEK via envelope encryption, key material in a KMS loaded only during a live session. See Infrastructure for the honest threat model.
  • CI on every PR — eslint, tsc --noEmit, tests, gitleaks, and the RLS grep.
