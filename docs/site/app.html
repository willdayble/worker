What the worker uses

The app the worker uses instead of WhatsApp.

Conversations come through WhatsApp and clients chat as normal, while the worker uses this app instead of the WhatsApp app. They see their conversations, move each one from enquiry to booked and done, and keep notes and safety flags as they go. These are well-understood CRM patterns; the research points to existing tools to borrow from rather than inventing anything new.

Getting set up

Workers should be able to see it work in a day or so. Setup is: create an account, connect a channel (scan a code or enter a pairing code, depending on the channel), watch recent conversations come in, and read a plain-language notice about what the AI does and what the app can and can't see.

What the notice covers

The notice covers both directions of AI use — drafting replies and analysing incoming messages — and says plainly that during the prototype the operator can technically decrypt data while a session is live. The consent is clear rather than buried.

Inbox to booking

Conversations come through WhatsApp, where clients chat as normal, while the worker uses this app instead of WhatsApp. Each conversation moves through a simple set of stages.

Figure 1 — The pipeline. Deliberately generic stages that any service business would recognise. Each contact sits in exactly one stage at a time.

  1. A message arrives. It appears in the inbox as a new enquiry, with the contact created automatically and a draft reply already suggested (the worker can ignore it).
  2. Screening. The worker asks their usual questions; an optional external screening check and an advisory red-flag signal help spot trouble early.
  3. Confirmed. When a time and the details are agreed, the booking is recorded — service, fee, date — in a couple of taps.
  4. Completed or Lost. Afterwards, the worker closes the loop so the system learns what actually happened.

Tagging & the services menu

Workers tag contacts and conversations with a customisable menu — the kinds of work offered, the intent of an enquiry, where in the pipeline things stand. The AI suggests tags from the message, but the worker owns the menu and the final say.

Neutral on the surface, always

Public-facing materials and the data model use general-purpose, neutral terms only. Whatever a worker calls their own services privately, the system's schema and anything an investor or the public sees stays generic. This is a deliberate legal and brand choice, not an accident of wording.

Recording how a booking ended

The most useful habit the app encourages is recording how each booking ended. Four outcomes cover most of what's worth knowing:

Won

Booked and completed. Service, fee and date logged — the basis for everything the dashboards show later.

Lost

Didn't convert, for ordinary reasons. Useful for seeing conversion.

Time-waster

Was never going to book. Recording it helps the worker spot the pattern next time.

Dangerous

Flagged for safety — advisory, set by a person, and kept separate from the channel so it survives a ban.

Because outcomes are recorded the same way each time, the cross-user dashboards (built for the test phase, with explicit consent) can show things a single worker couldn't see on their own: which sources convert, time-to-book, where time-wasters cluster.

Where clients came from

When a new client first gets in touch, the app records where they came from — which advert, link, or page — and writes it down once, at the point the contact is created. This is a standard first-touch attribution approach; the only thing that matters is writing it once and never overwriting it.

Tech info: how first-touch attribution is captured For technical readers

Standard marketing-attribution plumbing: capture UTM parameters into hidden form fields, persist them in a cookie across pages, and write the first-touch value into a write-once field at contact creation. The entry-point is stored per contact and never mutated, so attribution can't drift as a client interacts later.

The AI assistant

The AI helps; it doesn't act on its own. For each incoming message it does one analysis — is this a booking? what's the intent? any safety red flags? a suggested reply — and hands it all to the worker. The rules that decide what happens next live in ordinary code, not the model.

Two rules that don't change

Never auto-send. Never auto-block. A wrongly-sent message, or a false "time-waster" flag, has real consequences — a missed booking is lost income for the worker. So the AI only advises on incoming messages, and only drafts outgoing ones for the worker to approve.

Tech info: one structured call, and why disclosure runs both ways For technical readers

Each inbound message triggers a single schema-constrained model call that always returns valid structured data — booking flag and neutral booking fields, intent tags, service tags, advisory red flags, a suggested reply, and a confidence score. A confidence threshold in deterministic code decides whether to stage a draft or escalate to a human.

  • Neutral schema: field names and values stay general-purpose. The AI extracts generic appointment data; it does not broker a transaction.
  • Disclosure both directions: outbound — any AI-authored text reaching a client is approved by the worker (or clearly labelled as an assistant), to satisfy bot- disclosure laws. Inbound — clients' messages are AI-analysed for tagging and red-flagging, which is automated processing of someone else's sensitive data, so the red flags are advisory only and the privacy notice discloses the inbound analysis too.
  • Cost-aware routing: a fast, cheap model handles routine extraction with a large cached prompt; a stronger model is reserved for drafting and judgement. See Costs.

Safety

Safety runs through the whole app rather than sitting in one feature. An optional external screening source can be checked at the screening stage; an AI signal flags concerning patterns as a separate, advisory input; and every dangerous-client flag is set by a person, append-only, audit-logged, and stored separately from the messaging channel so a ban can't erase it. A person can undo a flag too, because a wrong flag has real costs.
